実装コードスニペット（GSAP 3 + Pixi.js 8）

● 単一カードの反転アニメーション（成功演出） – GSAPを用いた例。rotationYでY軸回転し、backアウトで軽い弾みを付けています。Pixi.jsの場合は3DではなくTextureの差替えになるため、ここではDOM要素版を示します。

// GSAP (GreenSock) を用いたカード反転
gsap.to(".card", {
  duration: 0.35,             // 所要時間 約350ms
  rotationY: 180,             // Y軸180度回転
  ease: "back.out(1.7)",      // オーバーシュートありのイージング（成功用）
  onStart: () => {
    playSound("success.wav"); // 回転開始時に効果音再生（必要なら）
  },
  onComplete: () => {
    // 裏面から表面に画像差し替え
    document.querySelector(".card").classList.add("revealed");
    // バイブレーション（成功: 単発短振動）
    navigator.vibrate?.(10);
  }
});


● 複数カードのカスケード反転 – GSAPのstaggerオプションで順次ずらして反転させます。各カードが0.15秒ずつ遅れて開始し、全体でリズミカルにめくれます。失敗演出ではオーバーシュートを抑えたeaseを使用。

// カード一覧を順番に反転（カスケード演出）
gsap.to(".card", {
  duration: 0.4,
  rotationY: 180,
  ease: "power2.out",       // 滑らかな減速（失敗時なのでバウンスなし）
  stagger: 0.15,            // 150ms間隔で次々開始
  onComplete: i => {
    // i番目のカード反転完了後の処理
    const cardEl = document.querySelectorAll(".card")[i];
    cardEl.classList.add("revealed");
    if(i === cards.length - 1) {
      // 最後のカードめくり終わり時にまとめて効果音・振動
      playSound("fail.wav");
      navigator.vibrate?.([15, 50, 15]); // 失敗: ダブル振動
    }
  }
});


● Pixi.jsでの低モーション版 – WebGL環境では立体回転できない場合もあるため、単にスケールとアルファでフェードインさせるフォールバック例です（prefers-reduced-motion検知時にも使用）。カードSpriteのtextureを成功/失敗に応じて差し替えつつフェード。

// Pixi.js: カードをフェード表示（低モーション環境用）
function revealCard(cardSprite, isSuccess) {
  // 成功/失敗に応じてテクスチャ変更
  cardSprite.texture = isSuccess ? successTexture : failTexture;
  cardSprite.alpha = 0;
  cardSprite.scale.set(0.8);  // 少し縮小状態から
  // GSAPでフェード + スケールアップ
  gsap.to(cardSprite, {
    duration: 0.2,
    alpha: 1,
    pixi: { scale: 1 },      // PixiPluginでspriteのscaleプロパティを補間
    ease: "easeOut",         // シンプルなイージング
    onComplete: () => {
      // 必要なら効果音・振動
      if(isSuccess) {
        playSound("success.wav");
        navigator.vibrate?.(10);
      } else {
        playSound("fail.wav");
        navigator.vibrate?.([15,50,15]);
      }
    }
  });
}

「やってはいけない」アンチパターン集

① 冗長すぎるアニメーション: 1回の演出が長時間（>0.5秒〜1秒）に及ぶとユーザ体験を阻害します
nngroup.com
(A)。特に頻繁に発生する操作に長いアニメは待ち時間のストレスとなり、イライラや離脱の原因になります。適切な長さに留め、ユーザが操作を待たされないようにします。

② 過度なバウンス・反復: 大きすぎるオーバーシュートや何度も弾む動きは目障りで不安定な印象を与えます
joshcollinsworth.com
(B)。UIアニメでは「やりすぎ」は禁物で、“控えめ”が基本です
joshcollinsworth.com
(B)。オーバーシュートはごく軽微にし、反復する場合も1回のバウンス程度に留めます（小さな弾みは◎、ポヨンポヨン跳ね続けるのは×）。

③ 同時に動かしすぎる: 一度に多数の要素をアニメーションさせるとユーザの視線が分散し、何に注目すべきか分からなくなります
moldstud.com
(C)。またデバイス負荷も高まりカクつきの原因に。
moldstud.com
が示すように、人は同時に動く対象が3つを超えると認知負荷が増大します。アニメ対象は絞り、順序付けて見せる（ステッガ表示）ことで解決します。

④ ユーザ操作をブロックする: アニメ再生中に次の操作を一切受け付けない設計はストレスになります。例えばカードをめくっている最中にユーザが別のUIを押せなくなると、「待たされ感」が生じます
nngroup.com
(A)。非同期処理でアニメと操作受付を両立させ、重要でない演出はユーザ操作があれば即スキップする設計にします。

⑤ “prefers-reduced-motion”を無視: ユーザが明示的に動き削減を求めているのに無視してアニメーションを流すのはアクセシビリティ違反です
w3.org
(A)。この設定を検知しない、または誤った実装（例えばno-preferenceをデフォルトで動的効果ONとみなす）は避けます。常にreduce指定を尊重し、代替表現（瞬時のフェード等）を提供します。

⑥ 低FPS・カクつきを放置: アニメが60fpsを維持できない状態でリリースすると、ユーザにはカクつきや操作遅延として知覚されます。特にモバイルの廉価端末や多重アニメ発生時にfps低下が起きやすいです
joshcollinsworth.com
(B)。対策せずに放置するのはNG。GPU活用やwill-changeの適切な使用でパフォーマンスを確保し、テストで問題が出た場合は演出簡略化・OFFのフォールバックを必ず入れます。

⑦ サウンド／バイブの過剰演出: 音量が大きすぎたり長すぎるサウンド、強力すぎるバイブレーションはユーザを驚かせたり不快にさせます。ネガティブなフィードバックに不協和音や大音量ブザーを当てるのも心理的負荷が高いです
ux.stackexchange.com
(B)。音と振動は必要最小限で十分であり、心地よい短さ・音量に留めます（「心地よいフィードバック音」をユーザは好み、不快音は避ける傾向
ux.stackexchange.com
）。特に誤答時に大袈裟なブザー音を鳴らすのは避け、静寂も選択肢に入れるべきです

調整ノブまとめ:

時間を変えれば演出の俊敏さや重厚感が変化（速め＝軽快、遅め＝ドラマチックだが諸刃）。

イージングを変えると印象が変化（直線＝機械的、減速＝自然、バウンス＝遊び心）。

オーバーシュート量で演出の派手さ調整（大きいほどカートゥーン的、小さいほど上品）。

カスケード遅延を変えると全体リズムが変化（短い＝素早い一斉公開感、長い＝一枚一枚見せるドラマ）。

音と振動の有無・強弱で演出のフィードバックの明確さ調整（強いほどリッチだがやり過ぎ注意）。

これらを製品固有のトーン＆マナーやユーザ層に合わせてチューニングすると、常に新鮮で心地よいカードリビール体験を提供できるでしょう。
